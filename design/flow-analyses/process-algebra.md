In the Unix filesystem model, a file consists of an inode (metadata and data block pointers) and one or more directory entries that reference that inode~\\cite{ritchie1974}. Removing any one name does not affect access through any other name.\n    57→\n    58→\\textbf{Immediate consistency.} A write through any hard link is immediately visible through all other hard links, because they share the same inode. There is no propagation delay, no synchronization protocol, no eventual convergence. Consistency is a structural property of the implementation, not a guarantee layered on top.\n    59→\n    60→\\textbf{Volume-local scope.} Hard links cannot cross mount points. This is a fundamental constraint, not an implementation limitation: the inode number that identifies a file is meaningful only within a single filesystem volume. Inode~42 on \\texttt{/dev/sda1} has no relationship to inode~42 on \\texttt{/dev/sdb1}. The mount point is the hard boundary of hard link scope.\n    61→\n    62→These properties are consequences of the same underlying mechanism: hard links are references to a shared inode within a single volume's inode table. The consistency is immediate because the reference is direct. The scope is volume-local because the inode table is volume-local. Unlike hard links, a symbolic link is a separate filesystem object---a small file whose contents are a pathname string. When the kernel encounters a symbolic link during pathname resolution, it substitutes the link's contents and continues resolution from there.\n    67→\n    68→Symbolic links changed the properties of linking in three important ways:\n    69→\n    70→\\textbf{Asymmetry.} A symbolic link has a defined direction: the link points to a target. Removing the target leaves a dangling link. Removing the link does not affect the target. The link and its target are not interchangeable, in contrast to hard links.\n    71→\n    72→\\textbf{Cross-boundary scope.} Because a symbolic link contains a pathname rather than an inode reference, it can refer to any path the kernel can resolve---including paths on other volumes, on NFS mounts, and on any filesystem the operating system has mounted. Coda's replication is per-volume, bidirectional (client-server), and orthogonal to file permissions. Flow links can be seen as a factoring of Coda's insight---that partition tolerance belongs in the filesystem---into a primitive that can be composed at a finer grain.\n   375→\n   376→\\subsection{Plan 9}\n   377→\n   378→The Plan~9 operating system, developed at Bell Labs beginning in the late 1980s~\\cite{pike1995}, introduced a namespace model in which each process has a private filesystem view constructed through \\texttt{bind} and \\texttt{mount} operations. These operations compose namespace from multiple sources, local and remote, into a unified hierarchy visible only to the process and its children.\n   379→\n   380→Plan~9's namespace composition is the closest prior art to flow links as a \\emph{namespace primitive}. The \\texttt{bind} operation---binding one directory tree over another---is structurally similar to declaring a relationship between two paths. Plan~9's insight that namespace should be a per-process construction, rather than a system-wide tree, is a direct precursor to the idea that cross-boundary relationships should be expressible in the filesystem.\n   381→\n   382→However, Plan~9's \\texttt{bind} and \\texttt{mount} operations are synchronous. They rely on the 9P protocol~\\cite{pike1993}, which requires the remote service to be reachable. A \\texttt{mount} operation to an unreachable server fails, just as an NFS mount does. Plan~9 advanced the composability of naming but did not address the partition-tolerant regime. It remains in the CP quadrant of the CAP tradeoff space---namespace composition with synchronous semantics.\n   383→\n   384→\\subsection{Ficus and Rumor}\n   385→\n   386→The Ficus filesystem, developed at UCLA in the early 1990s~\\cite{guy1990}, and its successor Rumor~\\cite{guy1998} implemented optimistic peer-to-peer file replication. Ficus maintained replicas across multiple servers, allowing updates at any replica and reconciling conflicts through version vectors and type-specific resolvers.\n   387→\n   388→Ficus and Rumor are notable for treating peer-to-peer replication (rather than client-server caching) as the normal mode of operation and for developing practical conflict detection and resolution mechanisms. The survey covers scheduling of propagation (eager vs.\\ lazy), conflict detection (syntactic vs.\\ semantic), conflict resolution (manual vs.\\ automatic), and numerous system architectures.\n   393→\n   394→The survey's taxonomy is thorough but frames replication exclusively as a \\emph{system-level} or \\emph{application-level} concern. The concept of a per-path, user-visible replication declaration---a primitive in the filesystem itself---does not appear in the taxonomy. This is not an oversight; it reflects the field's framing of replication as infrastructure rather than as something the filesystem should express directly. A reflink creates a copy of a file that shares the underlying data blocks with the original. When either copy is modified, the shared blocks are duplicated on demand (copy-on-write).\n   399→\n   400→Reflinks demonstrate that the link taxonomy is not closed: new link types can be defined and adopted by production filesystems. However, reflinks extend the \\emph{semantics} of linking within a volume (adding COW behavior) without extending the \\emph{scope} of linking across volumes. A reflink, like a hard link, is confined to a single filesystem volume. It occupies the same row of the scale ladder as hard links---partition-impossible, immediate consistency---with additional copy-on-write semantics.\n   401→\n   402→The existence of reflinks is relevant to our argument in two ways. It supports per-path synchronization rules, bidirectional propagation, and conflict detection. However, Unison is a tool, not a filesystem primitive. Its synchronization declarations live in its own configuration files, invisible to the filesystem. A directory listing shows no indication that Unison is synchronizing a path. No state reconciliation protocol is required. The destination consults its local store, identifies missing hashes, and requests them from any available source.\n   427→\n   428→\\textbf{Corruption detection.} Content that arrives with the wrong hash is immediately detectable. The identity check is built into the addressing scheme.\n   429→\n   430→\\textbf{Source independence.} Because content is identified by hash rather than by location, the destination can obtain content from \\emph{any} source that has it---the original sender, a peer, a CDN, a local cache, a shuttle drive. Hard links address intra-volume naming with immediate consistency. Symbolic links address intra-network naming with synchronous consistency. Neither addresses the partition-normal regime that characterizes much of modern computing: systems separated by organizational boundaries, air gaps, geographic distance, and intermittent connectivity.\n   511→\n   512→The CAP theorem provides the formal scaffolding for this argument. Now B has `x` from the first resolution but `y` from the second. These resolutions are inconsistent. Non-Determinism and Conflict\n\n### 8.1 Conflict as External Choice vs Internal Choice\n\nIn CSP, **external choice** (`[]`) offers the environment a choice between alternatives. **Internal choice** (`|~|`) makes the choice autonomously, without the environment's input.\n\nBidirectional flow link conflicts are *internal choice*: the system must choose between the two concurrent modifications, and the environment (the users at each end) has no say in the choice at resolution time. They had their say when they wrote -- the conflict arises precisely because both exercised their choice independently.\n\nThis maps to a well-known result in process algebra: **internal non-determinism reduces the set of properties a system can guarantee**. Specifically, a system with internal choice cannot guarantee any property that depends on *which* choice is made -- it can only guarantee properties that hold regardless of the choice.\n\n**Observation 13: The only properties that bidirectional flow links can guarantee unconditionally are those that hold for ALL possible conflict resolution outcomes.**\n\nFor example: \"at least one of the two writes is preserved\" can be guaranteed. \"Both writes are preserved\" can be guaranteed only with a merge-capable resolution (e.g., CRDT union). \"A specific write wins\" cannot be guaranteed without fixing the resolution function.\n\nThis is why the paper correctly identifies conflict resolution as a policy external to the primitive. The primitive introduces internal non-determinism; the policy resolves it. Fixing the policy in the primitive would eliminate useful behaviors (different applications need different resolutions).\n\n### 8.2 CRDTs as Deterministic Resolution\n\nCRDTs eliminate internal non-determinism by ensuring that all resolution outcomes are the same: the merge function is deterministic, commutative, associative, and idempotent, so the order of application does not matter.\n\nIn process algebra terms, a CRDT-based bidirectional flow link replaces internal choice with a **deterministic confluent merge**. The system still has multiple execution paths (different propagation orders), but all paths lead to the same quiescent state. This is **confluence** in the process algebra sense: all maximal computations yield the same result.\n\n**Observation 14: Content-addressed flow links with set-union merge are confluent.**\n\nThis is the strongest behavioral guarantee a bidirectional flow link can achieve: the quiescent state is independent of execution order. The pi-calculus model from Section 3 already showed that content-addressed flow creates independent per-item channels; confluence follows because independent channels do not interfere.\n\n---\n\n## 9.
